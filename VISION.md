# Loom Harness Vision Lock

This MVP is the first slice of a larger platform, not a scope reduction.

The target remains a compact multi-user online sandbox development platform with an auditable harness/loop:

- multi-user workspaces through Coder or `loomd`
- online sandbox development through browser IDE/preview links, terminal, persistent home, and isolated containers
- project/task isolation through worktrees and per-run event logs
- Gitea/Forgejo as the default truth layer for issues, PRs, comments, and code
- LiteLLM as the per-user model gateway and routing layer
- a first-party harness loop that records every user message/run comment, effective policy, model action, tool observation, verification, evaluator result, pause/resume, review, and finish decision
- `brain` as the compounding layer that learns from run summaries and proposes skill improvements through reviewable git changes
- human gates for merge, external side effects, permission escalation, and production deployment

## Non-Negotiable Design Constraints

1. The event schema must stay useful to a future web UI, audit view, Gitea comment bridge, and brain scoring job.
2. Tool execution must be isolated behind a runtime boundary so local execution can later become container or remote execution without changing the loop contract.
3. Verification and configured evaluation are gates, not decorations. A run cannot report success if either configured gate fails.
4. Human steering must become durable run/audit state, not transient browser-only chat.
5. The MVP may omit the web UI, but it must not erase the multi-user online sandbox target.
6. The system may start with scripted and command adapters, but the adapter contract must leave room for LiteLLM-backed model routing.

## Future Control Plane Option

`ngaut/agent-git-service` is a tracked candidate for the shared truth/control layer now that the harness, sandbox, and platform-readiness path have a serve-enabled provider seam. It should not replace the first-party harness/loop: the integration boundary is issues, comments, pull requests, Git transport, workspace branch lease evidence, agent identity, and optional wiki/memory surfaces. The run loop, tenant audit, workspace executor, human gates, VAS learning, and brain signal pipeline remain owned by this project.

Keep Gitea/Forgejo as the default near-term provider. Use `--control-plane-provider agent-git-service` to exercise the GitHub-compatible adapter path when we need agent-native APIs, durable agent accounts with scoped tokens, stronger issue/workspace semantics, or a provider-neutral control plane that can sit behind `gh`-style tooling.

## Execution Target

The current executors are local, Docker-backed, and Coder-SSH-backed. Coder runs now prepare the workspace by optionally creating it from a template, starting it, syncing the requested repo, and checking out either a task branch or an isolated per-run git worktree before the harness loop executes actions. Runs can now carry repo/branch/issue metadata, optionally comment final summaries back to Gitea/Forgejo issues, optionally create PRs, stop at a human review gate that supports approval/rejection/explicit merge, and stop at an admin deployment gate before production approval. Tenant policy now gives each tenant persistent keys, tenant-scoped model key envs, resource caps, workspace byte warnings/caps, tool allowlists, executor limit overrides, and an auditable admin-approved escalation path. Docker-backed harness execution, `loomd`-managed tenant containers, and the bundled Coder template now default to cap-dropped, no-new-privileges, read-only-rootfs sandbox profiles with bounded `/tmp`, resource caps, and explicit network boundaries while keeping the same `WorkspaceExecutor` contract; the HTTP `online-sandbox` profile readiness now rejects missing admin/developer/viewer tenant roles across bootstrap and policy-backed keys, local execution, and Docker-without-home as incomplete, and the local-executor safety gate treats existing policy-backed keys as tenant auth. `loomd serve` can host the periodic brain loop, optionally fast-forward sync the shared skills repo before scoring, push/open weak-skill proposal PRs, and stop idle tenant containers using host-side active-session markers without changing the per-tenant harness contract; `loomd ps` exposes those markers alongside Docker status for operator visibility.

The same isolation, idempotent-create, and model-routing contract is now machine-visible: server and tenant status expose `server.runWorkspaceIsolation`, `server.runCreateIdempotency`, active-run `workspaceLeaseScope`/`workspaceLeaseKey` details, and matching readiness checks; smoke emits server/tenant run isolation fields plus active-run lease evidence; and `platform-readiness` fails unless Coder uses run-scoped workspace isolation instead of the default project lock. Platform smoke also proves the same active run blocks online file writes, another completed run can still edit and create a run-scoped PR handoff while that same-project isolated run is active, omitted PR branches derive a runId-suffixed default branch, and peer smoke exercises cross-server active-run lease visibility, owner-loop run controls, plus simultaneous `clientRequestId` run-create replay. Running status readiness and doctor also require model routing to have a base URL plus server-wide, tenant-scoped, or policy API-key scoped model env coverage, exposing only non-secret key mode, tenant counts, and missing env names.
