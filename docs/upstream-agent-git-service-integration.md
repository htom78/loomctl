# Upstream agent-git-service Integration Backlog

Status: tracked candidate, not the harness/loop owner.

Reference: https://github.com/ngaut/agent-git-service

As of 2026-07-02, upstream describes `agent-git-service` as a self-hosted, GitHub-compatible API server for agents. The useful surfaces for us are REST v3, selected GraphQL v4, OAuth device flow, Git Smart HTTP, bare Git repository storage, TiDB/MySQL-compatible metadata storage, durable agent identities, scoped tokens, default workspaces, issue workspace features, and wiki memory.

The local integration entry is `scripts/upstream-agent-git-service-local.sh`. It pins upstream commit `9ab722e07b0797b67da05ecb72ad3c0feae6abd3`, builds upstream's own Dockerfile, requires the real TiDB/MySQL `DB_DSN` contract, waits for the real `/readyz`, and runs Loom's compatibility rehearsal in `candidateMode: upstream`. Its reports use `targetClass: local-upstream-e2e` and `externalStagingEligible: false`; they prove that Loom can talk to the real upstream implementation, but cannot satisfy the non-loopback external staging gate.

```bash
# Existing TiDB/MySQL-compatible DB_DSN
scripts/upstream-agent-git-service-local.sh fetch
scripts/upstream-agent-git-service-local.sh build
DB_DSN="$DB_DSN" scripts/upstream-agent-git-service-local.sh start
scripts/upstream-agent-git-service-local.sh wait
scripts/upstream-agent-git-service-local.sh verify
scripts/upstream-agent-git-service-local.sh stop

# Disposable one-command rehearsal using upstream's TiDB Zero quickstart API
LOOM_AGS_DATABASE_MODE=tidb-zero scripts/upstream-agent-git-service-local.sh e2e-zero
```

The script passes `DB_DSN` and the local AGS token by environment name, never writes either value to reports, refuses a dirty or wrong-revision source checkout, and only removes containers/volumes carrying its ownership label. `e2e-zero` removes its ephemeral container and Git volume after verification; the image and token-free evidence remain for inspection.

## Boundary

Keep these inside loom:

- harness loop, run lifecycle, pause/resume/cancel, idempotent create
- tenant audit, auth policy, model-key accounting, human gates
- Coder/local/docker executor isolation
- VAS-lite cases, learning projection, brain signals
- staging proof, operator bundle, secret scan, and cutover gates

Borrow these from upstream AGS only through the `control-plane-provider` seam:

- issue comments, signed webhooks, pull requests, merge/review evidence
- Git Smart HTTP clone/fetch/push URLs
- durable agent accounts and repo grants
- issue workspace presence/attachments
- wiki memory pages used by VAS learning projection

## Integration Steps

1. Run upstream `gh-server` as the candidate control plane.
2. Point `LOOM_AGENT_GIT_SERVICE_URL` at its external `/api/v3` URL and keep tokens in CI secret storage.
3. Generate a strict AGS bundle with `loom harness platform-cutover-plan --control-plane-provider agent-git-service --out plan.json --bundle-out cutover-bundle`.
4. Archive `external-secrets.json`, `github-actions-staging.yml`, and `upstream-agent-git-service.json` from the bundle; the first is the token-free CI secret-store env-name handoff, the second is a token-free GitHub Actions strict staging template with configurable `loom_bin`, `bundle_dir`, and optional source-tree bootstrap, and the third is the token-free handoff for `gh-server`, `/api/v3`, `DB_DSN`, Loom env names, the staged AGS issue/repo/wiki/write target, and the structured `operatorChecklist` that maps each upstream/loom phase to required evidence.
5. Run `loom harness platform-ci-handoff-preflight --dir cutover-bundle --repo-root <repo-root> --repo <owner/repo> --ref <branch> --report cutover-bundle/reports/ci-handoff-preflight.json` before the real Actions handoff, or write the same repo/ref with `loom harness platform-operator-github-actions-target-input --dir cutover-bundle --repo-root <repo-root> --repo <owner/repo> --ref <branch>` and pass the saved `cutover-bundle/github-actions-target.input.json` through generated `--target` command refs; it checks local bundle files, `external-secrets.json`, optional repo-root workflow install sha256, `gh` repo/workflow visibility from that repo root, and GitHub Actions required repository secret names without storing command output or secret values. If the configured GitHub CLI is missing, it reports `github.cli` before attempting downstream GitHub probes. If the workflow is not visible, run `loom harness platform-ci-workflow-publish-plan --dir cutover-bundle --repo-root <repo-root> --repo <owner/repo> --ref <branch> --report cutover-bundle/reports/ci-workflow-publish-plan.json` and execute its token-free git publish/verify command arrays. If required Actions secrets are missing, run `loom harness platform-ci-secrets-plan --dir cutover-bundle --repo <owner/repo> --report cutover-bundle/reports/ci-secrets-plan.json` and execute its token-free `secretSetShellCommands` from an operator shell that has the real env values.
6. Run `loom harness platform-staging-targets-plan --dir cutover-bundle --report cutover-bundle/reports/staging-targets-plan.json`, then either export the emitted `LOOM_REAL_*` target env names and run `applyCommandArgs`, or save a local `platform-staging-targets-input/v1` file through `loom harness platform-operator-real-staging-target-input --dir cutover-bundle --input <platform-staging-targets-input.json> --repo-root <repo-root> --repo <owner/repo> --ref <branch>`. Prefer `loom harness platform-operator-real-staging-targets-apply --dir cutover-bundle --auto-refresh-bundle --repo-root <repo-root> --repo <owner/repo> --ref <branch>` to apply that input, write `plan.real-targets.json` and `reports/real-staging-targets.json`, refresh the bundle from the real plan, and advance cockpit artifacts without printing values.
7. Run `loom harness upstream-agent-git-service-server-env-plan --dir cutover-bundle --report cutover-bundle/reports/upstream-agent-git-service-server-env-plan.json`, then run `LOOM_REQUIRE_EXTERNAL_STAGING=1 LOOM_CUTOVER_PHASE=pre-serve ./commands.sh`; strict `commands.sh` also writes the server env plan before `upstream-agent-git-service-handoff/v1`. The handoff report must prove the guide, `DB_DSN`, Loom env names, checklist, and secret scan, then `platform-staging-prerequisites/v1` must prove the operator shell has the strict bundle, env names, `LOOM_BIN`, `jq`, external targets, and upstream `DB_DSN` server-env name needed to attempt the run.
8. The AGS readiness stage must prove upstream `/readyz` reports `status: "ready"` before API/native-surface evidence can pass.
9. Require `agent-git-service-compat/manifest.json` to report `candidateMode: "upstream"` and a non-loopback candidate URL matching `staging-targets.json`.
10. Require `platform-staging-run/v1`, `staging-evidence`, `staging-verdict`, and `external-staging-audit.json` to pass before manual `serve`.
11. After `serve`, run AGS approval stages, `platform-operator-approvals`, final artifact verification, `platform-staging-proof`, `platform-goal-audit`, and final `platform-operator-status`.
12. Cut over only when `platform-goal-audit` passes and `operator-status.json` reports `phase: "production-cutover-ready"` with strict external operator artifacts, `platform-readiness`, multi-user runs, tenant audit, issue-comment handoff, VAS review, brain feedback, backup/restore dry-run, AGS native write, and no forbidden secret hits.

## Acceptance Gates

- `platform-preflight.json` proves external LiteLLM, Coder, and AGS targets.
- `external-secrets.json` is token-free, hash-anchored in the bundle manifest, lists required Loom env names only, and gives CI the `source ./env.sh` injection contract without exposing values.
- `github-actions-staging.yml` is token-free, hash-anchored in the bundle manifest, maps required Loom env names to GitHub Actions secrets by name, exposes `loom_bin` for installed binaries, `bundle_dir` defaulting to `cutover-bundle`, plus `node_version` and `bootstrap_source_tree` for building a checkout-backed wrapper through `actions/setup-node`, sets a same-ref/same-phase concurrency group with `cancel-in-progress: false`, runs the strict staging bundle commands while uploading reports from `${{ inputs.bundle_dir }}/reports`, records token-free `ci-run-proof.json` from GitHub Actions run metadata, and is installed at `.github/workflows/github-actions-staging.yml` with the same sha256 as the bundle template through `platform-ci-handoff-install`, whose archived `ci-handoff-install.json` must match both source and destination sha256. `platform-ci-workflow-publish-plan` turns the local workflow install into token-free git status/add/commit/push, repo/ref discovery, `gh workflow view`, and preflight command arrays without executing git or storing command output. `platform-ci-secrets-plan` turns `external-secrets.json` into token-free `gh secret set` shell commands using stdin so secret values do not appear in report JSON or `gh` argv. `platform-ci-handoff-preflight` checks local bundle readiness, `external-secrets.json`, optional repo-root workflow install sha256, GitHub CLI auth/repo/workflow visibility from the supplied repo root, and required GitHub Actions repository secret names before the external run without storing `gh` output, secret values, or unrelated repository secret names. `platform-ci-handoff-run` is the local audited orchestrator for install, preflight, dispatch, wait, artifact sync/import, and operator-status refresh; with an explicit `--repo-root`, it installs or refreshes the local workflow before preflight so the first failure can be the remaining remote GitHub visibility or secret gap. When `--ref` is omitted after a successful preflight, it carries the default branch from `gh repo view` into dispatch, run listing, waiting, and artifact sync. Workflow dispatch is audited locally with `platform-ci-workflow-dispatch`, which wraps `gh workflow run`, identifies the new run id through `gh run list` scoped to `workflow_dispatch` and the target ref, and writes `ci-workflow-dispatch.json` without command output; workflow waiting is audited with `platform-ci-workflow-wait`, which wraps `gh run watch --exit-status` and records parsed `gh run view` metadata without output. Actions report artifacts can be synced back into the local bundle with `platform-ci-artifact-sync` through `gh run download`, imported from an already downloaded directory with `platform-ci-artifact-import`, synced plus focused-imported as AGS pre/post-serve evidence with `platform-operator-ags-evidence-sync`, or imported from an already downloaded AGS artifact directory with `platform-operator-ags-evidence-import`.
- `upstream-agent-git-service.json` is token-free, hash-anchored in the bundle manifest, and structurally verified by `upstreamAgentGitServiceHandoffMismatches`, with upstream `gh-server` metadata, `DB_DSN` env name, Loom env names, the staged AGS target, and a complete `operatorChecklist` for upstream server, Loom env, pre-serve, serve, and post-serve evidence.
- `upstream-agent-git-service-server-env-plan.json` is token-free, generated from `upstream-agent-git-service.json`, contains required server env names, `serverStartCommandArgs`, `serverStartShellCommand`, and `/readyz` probe args, and does not read or persist `DB_DSN`.
- `upstream-agent-git-service-handoff.json` is token-free, has empty `missing`, passes guide, required-server-env, required-Loom-env, operator-checklist, and secret-scan gates, and is hash-anchored by final operator artifacts/proof.
- `staging-targets-plan.json` is token-free, generated from `plan.json`, lists placeholder LiteLLM/AGS/issue/repo/evidence targets, emits required `LOOM_REAL_*` env names, `envValidationCommandArgs`, `applyCommandArgs`, `inputTemplatePath`, `inputTemplate`, `applyInputCommandArgs`, `realTargetsCheckCommandArgs`, `validatedRealTargetsShellCommand`, and a `jq` patch command for `plan.real-targets.json`, and never reads or persists real target values. `platform-staging-targets-apply` writes the real plan and strict target proof while its own report remains token-free and contains only paths, input path/sha and `inputProof` when used, byte length, field names, sha256 values, gates, missing refs, next actions, and env names. Operator cockpit/status surfaces the ready input as `operator-real-staging-targets-apply`, backed by `platform-operator-real-staging-targets-apply --auto-refresh-bundle`, so the loop applies the input, refreshes the bundle, and refreshes cockpit artifacts in one audited command.
- `staging-targets-env-check.json` is token-free, generated from `LOOM_REAL_*` env presence and shape checks, rejects placeholder `.example` and sample repo refs, and does not print replacement target values.
- `staging-prerequisites.json` is token-free, has empty `missing`, and passes bundle, strict-command, env-name, tooling, external target, and upstream AGS handoff gates before any strict external pre-serve stage runs.
- `external-staging-audit.json` includes green bundle, environment, target, prerequisites, pre-serve evidence, staging-run, and verdict gates before reporting the pre-serve bundle ready for manual `serve`.
- `staging-run.json` is token-free strict external pre-serve proof with empty missing/forbidden lists and sha256 anchors for bundle verification, pre-serve env/run reports, staging targets, staging evidence, and verdict.
- `agent-git-service-compat/{manifest,baseline,candidate,compare}.json` are token-free and hash-anchored.
- `agent-git-service-contract-probe/v1` rejects empty 200 discovery shells: `/api/v3`, `/api/v3/meta`, and `/api/v3/rate_limit` must return minimal GitHub-compatible JSON shape, and failed shape checks are reported only as token-free `invalidEndpoints` plus per-endpoint `semanticErrors`.
- `agent-git-service-staging-readiness.json` proves `serverReadiness.ok`, `serverReadiness.status: "ready"`, and matching staged AGS base URL plus issue/repo/wiki target.
- `staging-evidence.json` is token-free, has empty missing/forbidden lists, passes external/preflight/AGS/compat gates, and hash-anchors staging targets, platform preflight, AGS staging readiness, and AGS compat artifacts.
- `agent-git-service-native-write-check.json` matches the staged write target, approved evidence URL, and wiki note sha256.
- `operator-approvals.json` matches `staging-ci.json.operatorApprovals[]`, selected stage, executed stage, gate id, and requirements.
- `ci-run-proof.json` is token-free, generated inside GitHub Actions, has `platform-ci-run-proof/v1`, successful run status, run URL, bundle workflow sha256, checkout-installed workflow sha256, optional install-report verification when present, and green run identity/install gates. Final cutover requires the post-serve proof; pre-serve proofs do not satisfy `operator-status.json` CI handoff readiness, proof `github.runId` must match the audited dispatch/wait run id when those reports exist, and the current `staging-proof.json` must hash-anchor that same CI proof.
- `ci-workflow-publish-plan.json` is token-free, generated locally from the bundle workflow plus repo root, contains installed workflow sha256 match, git status/add/commit/push args, repo/ref discovery args and field mapping, workflow verify args, and preflight args, and requires explicit `--repo`/`--ref` before it reports green.
- `ci-secrets-plan.json` is token-free, generated locally from `external-secrets.json`, contains required env names, `gh secret list` verification args, `gh secret set` args, and shell commands that pipe `${ENV}` through stdin without reading or persisting secret values.
- `ci-handoff-preflight.json` is token-free, generated locally before the handoff, has `platform-ci-handoff-preflight/v1`, records local bundle/workflow/external-secrets/staging-ci presence, optional repo-root workflow install status, optional `platform-ci-target-input/v1` path/sha256 for repo/ref handoff, sanitized `gh` command args, repo metadata, required/present/missing required secret env names, missing-secret `gh secret set` command args, and gates for CLI presence, auth, repo visibility, workflow visibility, GitHub Actions secret listing, required secret presence, and local workflow install when `--repo-root` is supplied without storing command output, secret values, or unrelated repository secret names.
- `ci-handoff-run.json` is token-free, generated locally when `platform-ci-handoff-run` orchestrates install, preflight, workflow dispatch, workflow wait, artifact sync/import, and operator-status refresh while preserving the individual subreports. Its top-level gates keep `artifactDownloadOk` and `artifactImportOk` separate so a failed GitHub artifact download is distinguishable from a downloaded artifact whose reports are missing, wrong-phase, wrong-run, or semantically invalid. Before refreshing `operator-status.json`, handoff-run writes a current `ci-handoff-run.json` snapshot so the status rollup can see the same handoff phase and run id it is reporting. With `--resume`, it reuses matching successful dispatch/wait reports only when they match the current phase and GitHub target repo/ref, so artifact sync can be retried without creating a duplicate Actions run.
- `ci-workflow-dispatch.json` is token-free, generated locally when `platform-ci-workflow-dispatch` wraps `gh workflow run`, records workflow, phase, inputs, command args, exit codes, and the newly identified run id/URL from `workflow_dispatch` runs on the requested ref without storing `gh` output.
- `ci-workflow-wait.json` is token-free, generated locally when `platform-ci-workflow-wait` wraps `gh run watch --exit-status`, records run id, phase, command args, exit codes, status/conclusion metadata, and no `gh` output.
- `ci-artifact-sync.json` is token-free, generated locally when `platform-ci-artifact-sync` calls `gh run download` for `loom-staging-reports`, records run id, phase, artifact name, download directory, command args, exit code, and nested import result without storing command output. When `--run-id` is omitted, sync chooses the audited workflow wait/dispatch run id before falling back to any existing `ci-run-proof.json`.
- `ci-artifact-import.json` is token-free, generated locally after downloading the GitHub Actions reports artifact, has `platform-ci-artifact-import/v1`, selected `phase`, optional expected run id, allowed report names from `staging-ci.expectedReports`, source/destination sha256 values for imported reports, and green present/JSON/copy gates. It rejects wrong-phase or wrong-run `ci-run-proof.json` and failed runner execute evidence before copying them into the bundle reports directory. `ags-evidence-sync.json` records the focused GitHub Actions download plus AGS import path, while `ags-evidence-import.json` is the focused AGS wrapper path generated by `platform-operator-ags-evidence-sync`, `platform-operator-ags-evidence-import`, or the browser cockpit sync/import controls. The import is refreshed into `operator-status.json.reports.agsEvidenceImport`, filters the selected phase to the AGS-specific expected-report allowlist, and refreshes cockpit plan/next plus `operator-handoff-packet.json` after importing server-env, handoff, readiness, compat, and post-serve native-write evidence.
- `operator-artifacts.json` anchors every `staging-ci.expectedReports` entry except its own final artifact/proof self-reference, including `staging-prerequisites.json`, `staging-run.json`, and `external-staging-audit.json`.
- `staging-proof.json` is strict external and strict approval proof, including token-free rollup status, empty missing/forbidden lists, operator artifact/post-serve/approval gates, hash-anchored `reports.stagingCiExpectedReports`, required-report `ok` checks, and current `staging-prerequisites.json`, `staging-run.json`, plus `external-staging-audit.json` ref/hash checks.
- `goal-audit.json` passes with `stagingRunProofAnchorOk`, `externalStagingAuditProofAnchorOk`, `operatorArtifactsPreServeReportsAnchorOk`, `operatorArtifactsStrictModeOk`, `operatorArtifactsCoreReportsAnchorOk`, `ciRunProofHandoffOk`, `ciRunProofStagingProofAnchorOk`, current operator artifact/approval proof anchors, confirmed cockpit runner execute lease evidence through `operatorCockpitRunnerExecuteOk` and `operatorCockpitRunnerProofAnchorOk`, AGS provider evidence, and `productionCutoverReady`.
- `operator-status.json` is token-free, generated after `goal-audit`, and reports `phase: "production-cutover-ready"` with empty `missing`, `goalAudit.ok: true`, strict external/approval/AGS flags, green pre-serve/serve/post-serve gates, `agentGitService.ok: true` with green server-env-plan, upstream-handoff, staging-readiness, and compat gates, AGS cockpit command refs for focused pre/post-serve `ags-evidence-sync` and `ags-evidence-import` plus handoff/readiness/compat/native-write while those gates are not green, and `ciHandoff` evidence for `github-actions-staging.yml`, workflow concurrency, `external-secrets.json`, required secret env names, `bundle_dir`, workflow install steps, `platform-ci-handoff-install` command args, installed workflow sha256 match for `.github/workflows/github-actions-staging.yml`, matching `ci-handoff-install.json` install-report evidence, matching `ci-run-proof.json` GitHub Actions run evidence, `platform-ci-handoff-preflight`, `platform-ci-handoff-run`, `platform-ci-workflow-dispatch`, `platform-ci-workflow-wait`, `platform-ci-artifact-sync`, and `platform-ci-artifact-import` command/report fields, `node_version`, the current `workflow_dispatch` phase, and copyable `gh workflow run` dispatch command fields. When final `goal-audit` is red because CI proof is missing or does not match the audited dispatch/wait handoff, the post-serve cockpit refs include `ci-handoff-run`, `ci-artifact-sync`, and `ci-artifact-import` before staging proof and goal audit reruns.
